# 기술 노트 (Technical Notes)

## 1. 외부 라이브러리 처리 방안
새로운 프로젝트에서는 '내가 작성한 코드'와 '외부에서 가져온 코드(라이브러리)'를 명확히 분리하여 관리하는 것이 중요합니다.

*   **`vendor.bundle.css` / `vendor.bundle.js`**: Swiper, AOS, jQuery 등 외부에서 가져온 라이브러리 파일들의 내용을 통합하여 이 파일에 저장합니다. 이는 우리의 커스텀 코드와 라이브러리 코드를 구분하고, 라이브러리 업데이트를 용이하게 합니다.
*   **`app.bundle.css` / `app.bundle.js`**: 사용자님께서 직접 작성할 모든 HTML, CSS, JavaScript 코드를 통합하는 파일입니다.
*   **HTML 로드 순서**: `index.html` 파일에서는 반드시 `vendor.bundle` 파일들을 먼저 로드하고, 그 다음에 `app.bundle` 파일들을 로드하여 라이브러리 기능이 먼저 준비된 후 우리 코드가 실행되도록 합니다.

## 2. '참고 자료 기반, 직접 코드 작성' 접근법
새로운 `index.html`을 구현할 때, 기존 `mental_health_portfolio/index-1.html` 또는 `mentalhealth.go.kr` 웹사이트를 '시각적 설계도'이자 '기능 명세서'로 활용할 것입니다.

*   **참고 자료 활용:** 실제 웹 페이지를 브라우저에서 보면서 페이지의 전체적인 모습, 각 섹션의 디자인, 기능 등을 파악합니다. 예를 들어 "헤더에는 로고, 메뉴, 검색 버튼이 있네", "슬라이더는 어떤 방식으로 작동하는구나"와 같이 분석합니다.
*   **직접 코드 작성:** 참고 자료의 소스 코드를 직접 복사-붙여넣기 하는 대신, 파악한 내용을 바탕으로 사용자님께서 직접 HTML, CSS, JavaScript 코드를 작성합니다. 저는 이 과정에서 올바른 방향으로 나아갈 수 있도록 가이드하고 필요한 개념을 설명해 드릴 것입니다.

## 3. 과거 기술 학습 교훈 (Lessons Learned from Previous Debugging)

기존 프로젝트를 진행하며 얻은 중요한 기술적 교훈들은 새로운 프로젝트를 더 효율적으로 구축하는 데 도움이 될 것입니다.

*   **CSS 파일의 '조용한 파싱 실패' 문제**:
    *   **현상:** 특정 CSS 파일(`portal_layout.css`)이 브라우저에서는 로드된 것으로 보이지만, 실제로는 스타일이 적용되지 않는 문제.
    *   **원인 추정:** 파일 내에 보이지 않는 특수 문자(BOM)나 인코딩 문제 등으로 인해 브라우저 파서가 파일을 정상적으로 해석하지 못했던 것으로 보입니다.
    *   **교훈:** CSS가 로드되었음에도 적용되지 않는 경우, 파일 내용뿐 아니라 파일 자체의 인코딩이나 보이지 않는 문자의 존재 가능성을 의심하고, 깨끗한 소스로 덮어쓰거나 파일을 새로 생성하는 것이 효과적일 수 있습니다.
*   **JavaScript `alert()`의 렌더링 차단 (Rendering Blocking) 현상**:
    *   **현상:** JavaScript에서 특정 요소에 CSS 애니메이션 클래스를 추가한 직후 `alert()`를 호출하면, 애니메이션이 시작되기 전에 `alert()`가 실행되어 애니메이션이 제대로 보이지 않는 문제.
    *   **해결책:** `alert()` 호출을 `setTimeout`으로 감싸 짧은 시간(예: 20ms) 지연시켜, 브라우저가 CSS 변경사항(애니메이션 시작)을 렌더링할 시간을 확보하도록 합니다.
    *   **교훈:** 브라우저의 렌더링 파이프라인을 이해하고, UI 변경과 JavaScript 실행 순서 간의 관계를 고려해야 합니다.
*   **SMIL vs. CSS Keyframe 애니메이션**:
    *   **배경:** SVG 애니메이션 구현 시 SMIL(`animate` 태그)을 사용했다가, CSS Keyframe으로 전환 후 다시 SMIL로 복귀하는 과정을 겪음.
    *   **교훈:** SMIL은 SVG와 애니메이션을 통합 관리하기 좋으나, 특정 상황(렌더링 차단 등)에서는 예상대로 동작하지 않을 수 있습니다. CSS Keyframe은 강력한 대안이지만, 상황에 따라 SMIL이 더 효율적일 수 있으므로 두 가지 방식의 장단점을 이해하고 적절히 활용해야 합니다.
*   **`[data-aos]` 속성과 `pointer-events` 문제**:
    *   **현상:** `aos.css`의 `[data-aos]` 속성이 적용된 요소의 자식 요소(`<a>`, `<button>` 등)가 클릭되지 않는 문제 발생.
    *   **원인:** `aos.css`가 `[data-aos]` 요소에 `pointer-events: none;`을 기본 적용했기 때문.
    *   **해결책:** 클릭되어야 할 자식 요소에 `pointer-events: auto;` CSS 규칙을 명시적으로 추가하여 부모의 설정을 덮어써야 합니다.
    *   **교훈:** 라이브러리를 사용할 때 해당 라이브러리가 DOM 요소에 어떤 CSS 속성을 적용하는지 이해하고, 필요한 경우 이를 재정의하여 문제를 해결해야 합니다.
